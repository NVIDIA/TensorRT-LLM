## Failure Groups

### TRTLLM-Gen FMHA static sparse kernel lookup

Failure:

- Several DeepSeek-V3.2 / GLM-5 FP4 MTP perf jobs failed with
  `Trtllm-gen kernels not found` for `headDimQk=576`, `headDimV=512`,
  `sparseType=1` on Blackwell.

Root cause:

- A previous conflict-resolution change disabled NVRTC fallback for all token
  sparse generation kernels.  That is too broad.  Dynamic token sparse kernels
  must still avoid NVRTC, but static sparse MLA generation needs NVRTC fallback
  when no prebuilt cubin exists for a shape such as `576x512`.

Resolution:

- Restored NVRTC fallback for static token sparse generation in
  `cpp/tensorrt_llm/kernels/trtllmGenKernels/fmha/fmhaKernels.h`.
- Kept the existing early return that disables NVRTC for dynamic token sparse.

### Kimi K2.5 EPLB setup

Failure:

- GB200/GB300 Kimi K2.5 EPLB jobs failed with
  `Cannot calculate num_local_slots. num_slots must be set and setup() must be called.`

Root cause:

- `maybe_create_moe_load_balancer()` checks the top-level HF architecture name.
  Kimi K2.5 uses `KimiK25ForConditionalGeneration`; this wrapper was missing
  from the EPLB model allowlist, so `MoeLoadBalancerConfig.setup()` was never
  called before MoE layer construction.

Resolution:

- Added `KimiK25ForConditionalGeneration` to `moe_model_arch_list`.
- Added a unit test that constructs a Kimi K2.5 `PretrainedConfig`, invokes
  `maybe_create_moe_load_balancer()`, and verifies `num_local_slots` is
  computed.

### Resource manager import fallout

Failure:

- `tests/unittest/_torch/executor` collection failed because
  `_warn_if_unsupported_v1_kv_cache_event_hash_algo` was imported but no longer
  existed in `resource_manager.py`.

Root cause:

- Rebase moved V1 KV-cache hash-algo validation logic, but the compatibility
  helper used by tests and warning paths was dropped.

Resolution:

- Reintroduced `_warn_if_unsupported_v1_kv_cache_event_hash_algo()`.
- The helper now accepts `auto` and `v1_block_key` silently, and warns for
  other algorithms before falling back to V1-compatible behavior.

### Single-GPU 2823 no-GPU collection failure

Failure:

- The test-list pipeline failed before stages could run because collection was
  executed with no visible CUDA device.  Import-time CUDA queries raised
  `RuntimeError: CUDA unknown error ... available devices to be zero`.

Root cause:

- Several modules and conftests queried CUDA device properties at import or
  collection time without first checking `torch.cuda.is_available()` or CUDA
  driver return codes.

Resolution:

- Guarded import-time CUDA queries in:
  - `tensorrt_llm/_torch/cuda_tile_utils.py`
  - `tensorrt_llm/_torch/custom_ops/torch_custom_ops.py`
  - `tensorrt_llm/_torch/attention_backend/flashinfer.py`
  - `tests/integration/defs/conftest.py`
  - `tests/integration/defs/triton_server/common.py`
  - `tests/unittest/conftest.py`
  - `tests/unittest/utils/util.py`
- No tests were skipped.  The guards only make collection/import degrade to
  "CUDA feature unavailable" when no device is visible.

### RoutingCustom / FP4 MoE CI failures

Failure:

- One multi-node perf job reported a `RoutingCustom.cu` launch failure.
- One multi-node perf job reported `Failed to initialize cutlass FP4 gemm`.

Local analysis:

- These are multi-node perf configurations and could not be reproduced directly
  on this single-node machine.
- The modified routing path was checked with the existing C++ routing test
  binary after rebuilding tests.
- CUTLASS+NVFP4 MoE was covered with the same small single-GPU parameter that
  appears in CI logs.

## B300 local follow-up

Question:

- Additional B300-local tests found failures in
  `tests/unittest/_torch/thop/parallel/test_fp4_gemm_quantize.py` and
  `tests/unittest/_torch/modules/moe/test_moe_backend.py`.
- Checked whether they were introduced by this PR / the final rebase-fix
  commit.

Analysis:

- `test_fp4_gemm_quantize.py` failures were all on the standalone
  TRTLLM-Gen GEMM path with `No kernel found for the given output type`.
  The final commit does not modify this test, the TRTLLM-Gen GEMM kernel
  directory, or the FP4 GEMM thop path. This is not a DSv4 PR regression.
- `test_moe_backend.py` CUTLASS+NVFP4 failures happened only during
  autotuner tactic replay. Normal module-level CUTLASS+NVFP4 paths passed.
  Root cause: on SM103, CUTLASS FP4 grouped MoE tactic enumeration included
  `NO_SMEM` epilogue schedules, but the C++ dispatch for block-scaled FP4 /
  finalize-fusion paths rejects `NO_SMEM` and requires TMA. The autotuner
  should not expose a tactic that the runtime dispatch cannot execute.

Resolution:

- Updated `cpp/tensorrt_llm/kernels/cutlass_kernels/cutlass_heuristic.cpp`
  so SM100/SM103 FP4 grouped MoE tactic generation filters non-TMA epilogue
  schedules, matching the runtime dispatch constraint. This is a real fix,
  not a test skip.
- After the tactic list shrank, a local rerun still failed when
  `/tmp/moe_autotuner_cache.json` contained stale tactic id `81` from the
  previous build while the rebuilt runner exposed only `70` tactics. Updated
  `tensorrt_llm/_torch/autotuner.py` to validate cache-hit tactics against the
  current runner's `get_valid_tactics()`. Stale entries are dropped and
  re-profiled instead of being replayed into C++ and failing with
  `vector::_M_range_check`.
